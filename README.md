# Summit booking portal

The client-facing booking page for **Summit Casino Events** — a client opens one link,
views their quote, e-signs the Event Booking Agreement, and pays the deposit (later, the
balance) via Stripe, without leaving the brand. Hosted on Cloudflare (Worker + D1).

This is a **standalone deployable**, deliberately separate from the private Summit desktop
app (`SmokinW33D/summit-ops`). The two talk only over HTTPS at runtime; neither imports the
other. Keeping the portal in its own repo means the CRM's code is never connected to
Cloudflare.

## What is and isn't in this repo

- **In it:** the booking-page code only — HTML/JS page, the API, the D1 schema.
- **Not in it:** no client data, no invoices, no contracts, no Stripe keys, no passwords.
  Client data lives in the D1 database (on Cloudflare) and in the desktop app; secrets live
  in Cloudflare's encrypted store. You could make this repo public and leak nothing about a
  client — but keep it **private** anyway.

## Security posture

- Booking links are **unguessable** (192-bit random tokens), **expire** (default 14 days),
  and each booking is **purged** the moment it completes — the store only ever holds a
  handful of in-flight bookings, never a growing history.
- No endpoint lists all bookings without the desktop's secret key; a client can only read
  their own booking, and only with their link. Card data goes browser→Stripe directly and
  never touches this server (PCI SAQ-A). The whole subdomain is `noindex`.

## Configuration

Three secrets, set on Cloudflare (dashboard → Worker → Settings → Variables and Secrets, or
`wrangler secret put`): `STRIPE_SECRET_KEY`, `STRIPE_WEBHOOK_SECRET`, `DESKTOP_API_KEY`
(the last is generated by the Summit app). Plus a D1 binding named `DB`. The Stripe
publishable key and portal URL live in the Summit app, not here — see `docs/PORTAL.md` in
`summit-ops` for the full runbook.

## Develop / deploy

```
pnpm install
pnpm run check      # typecheck
pnpm test           # pure-logic tests
pnpm run dev        # wrangler local (needs .dev.vars — copy from .dev.vars.example)
pnpm run deploy     # wrangler deploy  (or connect this repo in the Cloudflare dashboard)
```

The Worker creates its own D1 tables on first request — there is no migration step.
