# Summit booking portal

The client-facing booking page for **Summit Casino Events** — a client opens one link,
views their quote, e-signs the Event Booking Agreement, and pays the deposit (later, the
balance) via Stripe, without leaving the brand. Hosted on Cloudflare (Worker + D1).

This is a **standalone deployable**, deliberately separate from the private Summit desktop
app (`SmokinW33D/summit-ops`). The two talk only over HTTPS at runtime; neither imports the
other. Keeping the portal in its own repo means the CRM's code is never connected to
Cloudflare.

## Deploy your own (one click)

[![Deploy to Cloudflare](https://deploy.workers.cloudflare.com/button)](https://deploy.workers.cloudflare.com/?url=https://github.com/SmokinW33D/summit-portal)

Clicking the button makes Cloudflare **copy this repo into your own account, create a fresh
D1 database for you, and prompt you for the three secrets** — you edit nothing. Each person
who deploys gets their **own** isolated instance (own database, own keys, own address);
nobody shares anyone else's. After it deploys:

1. Add your booking domain (Worker → Settings → Domains & Routes → `book.yourdomain.com`).
2. In Stripe, add a webhook to `https://<your-domain>/api/stripe/webhook` and paste its
   signing secret into the Worker's `STRIPE_WEBHOOK_SECRET`.
3. In the Summit app → Settings → Client portal: enter the URL + Stripe publishable key,
   **Generate app key** (that's your `DESKTOP_API_KEY`), Save, Test connection.

(The button needs read access to this repo. It's yours now; make it **public** if you want
others to be able to click it — safe, since the repo holds no data and no secrets.)

## What is and isn't in this repo

- **In it:** the booking-page code only — HTML/JS page, the API, the D1 schema.
- **Not in it:** no client data, no invoices, no contracts, no Stripe keys, no passwords.
  Client data lives in the D1 database (on Cloudflare) and in the desktop app; secrets live
  in Cloudflare's encrypted store. You could make this repo public and leak nothing about a
  client — but keep it **private** anyway.

## Security posture

- Booking links are **unguessable** (192-bit random tokens), **expire** (default 14 days),
  and each booking is **purged** the moment it completes — the store only ever holds a
  handful of in-flight bookings, never a growing history.
- No endpoint lists all bookings without the desktop's secret key; a client can only read
  their own booking, and only with their link. Card data goes browser→Stripe directly and
  never touches this server (PCI SAQ-A). The whole subdomain is `noindex`.

## Configuration

**Two** secrets, set on Cloudflare (Worker → Settings → Variables and Secrets):
`STRIPE_SECRET_KEY` and `DESKTOP_API_KEY` (generated by the Summit app). The Worker
**auto-registers its own Stripe webhook** and stores the signing secret in D1 on first
publish, so `STRIPE_WEBHOOK_SECRET` is not needed in production. Plus a D1 binding named
`DB`. The Stripe publishable key and portal URL live in the Summit app, not here — see
`docs/PORTAL.md` in `summit-ops` for the full runbook.

## Develop / deploy

```
pnpm install
pnpm run check      # typecheck
pnpm test           # pure-logic tests
pnpm run dev        # wrangler local (needs .dev.vars — copy from .dev.vars.example)
pnpm run deploy     # wrangler deploy  (or connect this repo in the Cloudflare dashboard)
```

The Worker creates its own D1 tables on first request — there is no migration step.

## Keeping the live instance in sync

The portal that's actually **deployed** is a separate repo — `SmokinW33D/summit-booking-portal`
— created by the Deploy button. It's identical to this template except for its `wrangler.toml`
(which carries the D1 `database_id`). This template is the source of truth for the code; after
any change here, mirror it to the live instance with:

```
bash scripts/sync-instance.sh
```

That copies everything except `wrangler.toml`, commits, and pushes — and Cloudflare
auto-deploys the instance in ~1–2 minutes. Running it is the ONLY supported way to update the
live portal, so the two repos can't drift. (A future upgrade could do this automatically via a
GitHub Action on push, using a token with write access to the instance.)
